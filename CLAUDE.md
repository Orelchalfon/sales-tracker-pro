## Testing After Every Feature

After implementing any feature, **manually verify the full interactive flow** by starting the dev server (`pnpm dev`) and exercising it end-to-end in the browser:

- **Add sale**: Fill SalesForm → submit → confirm row appears in SalesTable and SummaryCards update
- **Edit sale**: Click edit on a row → modify fields → save → confirm table reflects changes
- **Delete sale**: Delete a row → confirm it's removed and summary totals recalculate
- **Filter by month**: Change MonthSelect → confirm only matching sales are shown and cards update
- **Export**: Click ExportButton → confirm `.xlsx` downloads with correct Hebrew content, RTL formatting, and totals
- **Persistence**: Add data → refresh page → confirm localStorage restored state correctly

Test the happy path and edge cases (empty state, single sale, boundary month transitions).

## Stack

- **Framework**: TanStack Start (React 19, SSR-capable) with file-based routing via TanStack Router
- **Data layer**: `useSales` hook with localStorage (key: `"sales-tracker-v1"`) — no backend, no server functions
- **Styling**: Tailwind CSS v4 + Radix UI primitives (shadcn/ui via `components.json`)
- **Forms**: React Hook Form + Zod
- **Export**: `xlsx` library — runs client-side only; do not move to a server function
- **Deployment**: Cloudflare Workers (`wrangler.jsonc`; entry: `src/server.ts`)

## Architecture

Single-route Hebrew RTL sales tracker. The root layout (`src/routes/__root.tsx`) sets `<html lang="he" dir="rtl">` and mounts the `QueryClientProvider`. The sole route (`src/routes/index.tsx`) composes all feature components.

**Core data flow:**
1. `useSales()` (`src/hooks/useSales.ts`) owns all CRUD state and localStorage sync
2. `SalesForm` adds sales → `SalesTable` displays/edits/deletes → `ExportButton` exports via XLSX
3. `MonthSelect` drives the active filter; `SummaryCards` aggregates the filtered set

**Domain type** (`src/types/sale.ts`):
```typescript
type Sale = {
  id: string;
  saleDate: string;     // YYYY-MM-DD — parse with parseSaleDate(), never new Date("YYYY-MM-DD") (UTC bug)
  systemName: string;
  systemPrice: number;  // ILS
  commission: number;   // ILS
  createdAt: string;    // ISO timestamp
};
```

**Server-side plumbing** (`src/start.ts`, `src/server.ts`, `src/lib/error-capture.ts`) is standard TanStack Start + Cloudflare Workers boilerplate. Never introduce server functions — there is no server-side state.

`src/routeTree.gen.ts` is auto-generated by TanStack Router — never edit it manually.

## Conventions

**Routing**: Add new routes as files under `src/routes/`. Add `head()` metadata in each new route file (root metadata lives in `__root.tsx`). MVP scope is a single `/` route — only add more if explicitly requested.

**Styling**: Use only semantic tokens (`bg-card`, `text-foreground`, `text-primary`, `bg-muted`, etc.) — never raw colors like `bg-white` or `text-black`. Add new tokens in `src/styles.css` using `oklch`. Keep `--primary` dark navy (current value) for contrast on the commission card.

**Dates**: Always use `parseSaleDate()` and `getSaleYearMonth()` from `src/types/sale.ts`. `new Date("YYYY-MM-DD")` parses as UTC midnight and shifts the day in local time.

**Currency**: Always `formatILS()` — never raw `.toLocaleString()` calls.

**RTL**: Global via `<html dir="rtl">` in `__root.tsx`. Do not add `dir="rtl"` per-component.

**Validation**: Validate unknown data with `isSale()` (or the Zod schema if added) before trusting localStorage reads.

## UI Components

`src/components/ui/` — shadcn/ui primitives (~47 components, already installed). `src/components/sales/` — domain-specific feature components. Add new UI primitives to `ui/`, new feature components to `sales/`.

## Localization

All user-facing strings are in Hebrew. Month names and date formatting use the `he-IL` locale.
